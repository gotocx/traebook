# TraeBook.skill

> 目标：让用户用“像聊天一样简单”的方式输入需求，系统用“像课程设计一样复杂”的机制，持续产出**同一视觉/信息风格**的可拔插电子书（入口页 + 章节页 + 练习页 + 分享链接），并且能覆盖多个品类（不仅是数据结构/算法）。

## 1. 这不是“写作器”，是“教学编排器”

TraeBook 的核心不是“生成文字”，而是做 **学习路径设计**：

- 先建知识结构（概念层级、依赖关系、学习目标）
- 再做章节编排（由浅入深，控制认知负荷）
- 再做解释与练习闭环（理解 → 操作 → 检验 → 纠错 → 迁移）
- 最后做发布与分享（可拔插网页、二级章节、可复现的分享状态）

## 2. 输出形式（统一：可拔插插件书）

输出必须包含：

1) `book/manifest.json`：书籍目录与路由清单  
2) `book/index.html`：入口页（目录 + 推荐学习路线）  
3) `book/chapters/<id>.html`：章节页（统一组件风格）  
4) `book/assets/`：配色/字体/图标/插图（可选）  
5) `book/share/`：分享数据（state json / share code，可选）

推荐集成方式：
- 对外只暴露一个 `<trae-book>` 或 `<algo-book>` 组件入口
- 通过 `page=` / `chapter=` / `topic=` 属性或路由参数加载指定章节

## 3. “用户很简单、机制很复杂”的交互协议

TraeBook 采用三层协议：

### A. 轻量入口（用户只说一句话）
用户只要说：  
> “我想学习 X，给我做一本书”

系统默认启动：`Discovery → Map → Outline → Chapter 1` 流程。

### B. 深度机制（系统内部必须做的事情）
每一章生成前必须完成：

1) **学习者画像（最低可用）**  
   - 目标：为了考试/工作/兴趣/面试/教学  
   - 基线：0基础/有编程经验/科班/非科班  
   - 可用时间：每次学习时长、总周期  

2) **知识地图（Concept Graph）**  
   - 概念节点（定义、直觉、形式化、常见误区）  
   - 依赖边（先学什么才能学什么）  
   - 每章覆盖哪些节点  

3) **章节结构（Chunking）**  
   每章固定结构，避免“随便写一章”：
   - 学习目标（3条以内）
   - 直觉解释（不超过 200 字）
   - 正式定义（必要时）
   - 示例 1（最小示例）
   - 反例/误区（至少 1 个）
   - 练习 1（带提示）
   - 练习 2（迁移题）
   - 本章总结（可复述）
   - 下一章桥接（为什么要学下一章）

4) **深度解释要求（3 层解释）**
同一知识点必须至少有两层（建议三层）：
1. 直觉层：比喻/图景/类比  
2. 规则层：步骤/规则/条件  
3. 工程层：实现细节/边界/复杂度/常见坑

### C. 逐章节奏（不要一口气“写完整本书”）
默认节奏：
1. 先产出：目录 + 第一章（可运行/可分享）  
2. 用户反馈：是否懂、是否快、是否难  
3. 再产出：第二章 + 练习页  
4. 每 3 章做一次：复盘/测验/补缺口

> 硬规则：一次对话只产出一章，详见 `constraints.md`。

## 4. 覆盖多品类的方式（主题无关框架）

TraeBook 的框架与主题无关，关键是“知识结构”。  
支持的品类示例：

- 编程：语言入门、工程化、测试、性能、架构
- 产品：需求分析、竞品、PRD、用户研究
- 设计：设计系统、交互原则、信息架构
- 业务：行业入门、流程、合规、风控
- 源码解读：项目结构、入口、运行时、权限、插件

不同品类只需要换“知识地图模板”和“练习类型模板”。

## 5. 质量闸门（必须执行）

每章发布前必须过 `质量清单`：

- 可复述：读者能用自己的话复述（章末 3 问）
- 可操作：至少 1 个交互/步骤化例子（哪怕是文字模拟步骤）
- 有误区：至少 1 个“你可能会误解成…但其实…”
- 有练习：至少 2 题，其中 1 题迁移
- 有桥接：本章与下一章的因果关系明确

详见：`checklists/quality.md`
